\section{Introduction}

SQL, or Structured Query Language, is a  declarative, domain-specific programming language designed to query, manipulate, and transform data from a relational database. A \emph{relational database} represents a collection of related (two-dimensional) tables, each with a fixed number of named columns (the attributes or properties of the table) and any number of rows of data. The database \emph{schema} describes the structure of each table, and the datatypes that each column of the table can contain.

All popular \emph{relational database management systems} (RDBMSs) -- SQLite, MySQL, Postgres, Oracle and Microsoft SQL Server -- support SQL. Because the SQL standard is large, complex, does not specify the database behavior in some important areas, defers some decisions to individual implementations, and is ambiguous for some construct semantics, all these RDBMS vendors use different implementations of SQL and therefore differ slightly in syntax, ruleset, and the kind of operators and functions they support. In particular, date and time syntax, string concatenation, \mintinline{sql}{NULL}s, and comparison case sensitivity vary from vendor to vendor. PostgreSQL and Mimer SQL strive for standards compliance, though PostgreSQL does not adhere to the standard in all cases. The most popular commercial and proprietary RDBMSs are Oracle (whose \mintinline{sql}{DATE} behaves as \mintinline{sql}{DATETIME} and lacks a \mintinline{sql}{TIME} type) and MS SQL Server (before the 2008 version). As a result, SQL code can rarely be ported between database systems without modifications.

SQL may be informally partitioned into sublanguages, which are commonly: data query language (DQL), data manipulation language (DML), data definition language (DDL), and data control language (DCL). Each SQL statement can be broadly classified as belonging to one of these, and we will use this classification.

\section{Data query language}

A \emph{data query language} (DQL) are for performing queries on a database. SQL supports this with the \mintinline{sql}{SELECT} statement. DQL is often considered a part of a data manipulation language (DML), i.e., the \mintinline{sql}{SELECT} statement can also be considered a DML statement.

\subsection{\mintinline{sql}{SELECT}}

To \emph{query} a database, use the \mintinline{sql}{SELECT} and \mintinline{sql}{FROM} clauses, which yields a table:

\begin{minted}{sql}
SELECT col, another_col, ...
FROM table;
\end{minted}

You can use \mintinline{sql}{*} to select all columns. Capitalizing commands is simply a convention that makes queries easier to read, but isn't enforced. The SQL standard defines that statements end with the semicolon statement terminator, though it's not required on every platform. Also, SQL treats one space, multiple spaces, or a line break as being the same thing. Columns names are typically lower case, and use underscores instead of spaces, because else you need to use double quotes. You can define aliases for column and table names in your result with the \mintinline{sql}{AS} keywords, although this keyword is optional and the SQL standard lets you use aliases with or without \mintinline{sql}{AS}, but without \mintinline{sql}{AS}, you cannot use a PostgreSQL keyword as alias. You can also use expressions to transform raw column values in a query with mathematical/arithmetic and string functions, and combine different column values (within the same row):

\begin{minted}{sql}
SELECT col AS Column, another_col AS "Another Column",
       (some_numerical_col + another_numerical_col) / 2.0 AS avg_col
FROM long_table_name AS table;
\end{minted}

You can round floating point numbers with \mintinline{sql}{ROUND(num, <precision>)}, e.g. \mintinline{sql}{ROUND(num, 2)} for rounding to three decimals. You can get the length of a string with \mintinline{sql}{CHAR_LENGTH(content)}. The \mintinline{sql}{DATEDIFF(date1, date2)} function returns the difference between two dates as an integer. You can convert between datatypes in queries using two types of syntax that do the same: \mintinline{sql}{CAST(column AS integer)} or \mintinline{sql}{column::integer}.

You may specify multiple comma-separated table names in the \mintinline{sql}{FROM} clause (and you may specify the same name multiple times, though you need aliases in that case), which will result in a Cartesian product of the given tables, i.e. every row from any table is paired with every row from any other table.

You can discard duplicate rows (taking only the values in the specified column(s) into account) by using the \mintinline{sql}{DISTINCT} keyword (you add one \mintinline{sql}{DISTINCT} keyword behind \mintinline{sql}{SELECT}, so not per individual column name), so you get all unique combinations of values across those columns:

\begin{minted}{sql}
SELECT DISTINCT column, another_column, ...
FROM mytable
\end{minted}

\subsection{\mintinline{sql}{LIMIT}}

The less results returned, the faster a query runs. The \mintinline{sql}{LIMIT} clause will reduce the number of rows to return, and the optional \mintinline{sql}{OFFSET} clause will specify where to begin counting the number rows from (useful for pagination of results):

\begin{minted}{sql}
SELECT column, another_column, ...
FROM table
LIMIT 100 OFFSET 200;
\end{minted}

\subsection{\mintinline{sql}{WHERE}}

To filter results, use a \mintinline{sql}{WHERE} clause with one or more conditions (concatenated with \mintinline{sql}{AND} or \mintinline{sql}{OR}) that are applied to each row:

\begin{minted}{sql}
SELECT column, another_column, ...
FROM table
WHERE
/* Numerical conditions */
      column >= 4 -- Standard numerical operators are supported
  AND column BETWEEN 1.5 AND 10.5 -- Number is within range of two values (inclusive)
  AND column >= 1.5 AND column <= 10.5 -- Same thing
  AND column NOT BETWEEN 1 AND 10 -- Number is not within range of two values (inclusive)
  AND column IN (2, 4, 6) -- Number exists in a list
  AND column NOT IN (1, 3, 5); -- Number does not exist in a list
/* Textual conditions */
  AND column = "abc" -- Case sensitive exact string comparison (notice the single equals)
  AND column != "abcd" -- Case sensitive exact string inequality comparison
  AND column LIKE "ABC" -- Case insensitive exact string comparison
  AND column NOT LIKE "ABCD" -- Case insensitive exact string inequality comparison
  AND column LIKE "%AT%" -- % matches a sequence of zero or more characters (e.g. "AT", "ATTIC")
  AND column LIKE "AN_" -- _ matches exactly one character (e.g. "AND", but not "AN")
  AND column IN ("A", "B", "C") -- String exists in a list
  AND column NOT IN ("D", "E", "F") -- String does not exist in a list
\end{minted}

In some variants of SQL \mintinline{sql}{LIKE} is case-sensitive, and \mintinline{sql}{ILIKE} is case-insensitive.

Full-text search is best left to dedicated libraries like Apache Lucene or Sphinx.

\subsection{\mintinline{sql}{ORDER}}

You can sort your results by one or more columns in ascending (\mintinline{sql}{ASC}, which is the default, so you can omit it) or descending (\mintinline{sql}{DESC}) order using the \mintinline{sql}{ORDER BY} clause. The results are sorted first by the column first specified, then the second, etc.:

\begin{minted}{sql}
SELECT column, another_column, ...
FROM table
ORDER BY column DESC, another_column ASC, ...;
\end{minted}

\subsection{\mintinline{sql}{JOIN}}

Databases are often broken down into pieces and stored across multiple orthogonal tables using a process known as \emph{normalization}. The term "relational database" refers to the fact that the tables within it "relate" to one another; they contain common identifiers (\emph{keys}) that allow information from multiple tables to be combined easily. This minimizes duplicate data in any single table, and allows for data in the database to grow independently of each other. Tables that share information about a single entity need to have a \emph{primary key} -- a column -- that identifies that entity uniquely across the database. \mintinline{sql}{JOIN} clauses in a query combine row data across two separate tables using this unique key. \mintinline{sql}{INNER JOIN} matches rows from the first table and the second table which have the same key (as defined by the \mintinline{sql}{ON} constraint) to create a result row with the combined columns from both tables. \mintinline{sql}{INNER JOIN} is the default, so you can use \mintinline{sql}{INNER JOIN} and \mintinline{sql}{JOIN} interchangeably, but it's more readable if you write it out. A \mintinline{sql}{LEFT JOIN} includes all rows from the table in the \mintinline{sql}{FROM} clause regardless of whether a matching row is found in the table in the \mintinline{sql}{LEFT JOIN} clause. Vice versa for \mintinline{sql}{RIGHT JOIN}. In a \mintinline{sql}{FULL JOIN}, rows from both tables are kept, regardless of whether a matching row exists in the other table. You might see queries with \mintinline{sql}{LEFT/RIGHT/FULL OUTER JOIN}, but the \mintinline{sql}{OUTER} keyword is really kept for SQL-92 compatibility and is usually left out. There is also \mintinline{sql}{CROSS JOIN} for the Cartesian product, and this is the same as specifying multiple tables in the \mintinline{sql}{FROM} clause. It has no \mintinline{sql}{ON} clause because you're just joining everything to everything.

\begin{minted}{sql}
SELECT column, another_table_col, ...
FROM table AS t
    INNER/LEFT/RIGHT/FULL JOIN another_table AS at ON t.id = at.id AND at.id > 5;
\end{minted}

Conditions in the \mintinline{sql}{WHERE} clause are applied to the joined tables after the join, but you can filter one or both of the tables before joining them and create matches between the tables only under certain circumstances by adding additional conditions after \mintinline{sql}{ON}, as above. You can also match on multiple foreign keys, which is mainly used for complex performance reasons.\\
You may also join a table with itself, e.g. by matching every row on a date column with a row from a day before.

When working with multiple tables, column names across tables may have the same name. As demonstrated above, you can always prefix column names with the corresponding table or alias names, which is often necessary to prevent ambiguity errors, e.g. \mintinline{sql}{t.id} or \mintinline{sql}{table.*}.

\mintinline{sql}{NULL}s indicate the absence of a value. When using \mintinline{sql}{LEFT/RIGHT/FULL JOIN}, you will likely have to write additional logic to deal with \mintinline{sql}{NULL}s in the result and constraints. An alternative to \mintinline{sql}{NULL}s is to have data-type appropriate default values in your database, e.g. 0 for numerical data, empty strings for text data, etc. You can test a column for \mintinline{sql}{NULL} values in a \mintinline{sql}{WHERE} clause with \mintinline{sql}{IS (NOT) NULL}. The \mintinline{sql}{IFNULL(expression, alt_value)} function is useful for dealing with \mintinline{sql}{NULL}s, which simply returns the given \mintinline{sql}{expression} if it isn't \mintinline{sql}{NULL}, and \mintinline{sql}{alt_value} if it is, e.g. \mintinline{sql}{IFNULL(maybenull, 0)}.

You can also concatenate multiple (different types of) \mintinline{sql}{JOIN}s. \mintinline{sql}{JOIN}s are left associative, i.e. each subsequent join joins a new table to the derived table that is the result of all the joins before it.

\subsection{\mintinline{sql}{GROUP}}

You can use aggregate expressions (or functions) that allow you to summarize information about groups of rows, i.e. vertically, across rows:

\begin{minted}{sql}
SELECT AGG_FUNC(column_or_expression) AS aggregate_description, ...
FROM table
GROUP BY column1, column2, ...;
\end{minted}

The \mintinline{sql}{GROUP BY} clause works by grouping rows that have the same value(s) in the column(s) specified (so the order of column names doesn't matter, contrary to \mintinline{sql}{ORDER BY}). Each aggregate function runs on each group, so you get as many results as there are unique groups/values in the specified column. Without a specified grouping, each aggregate function is going to run on the whole set of result rows and return a single value. \mintinline{sql}{AGG_FUNCTION} can be any of \mintinline{sql}{COUNT} (counts the number of rows with non-\mintinline{sql}{NULL} values in the specified column; you can also use \mintinline{sql}{COUNT(*)} to count all rows), \mintinline{sql}{MIN}, \mintinline{sql}{MAX}, \mintinline{sql}{AVG} (ignores \mintinline{sql}{NULL}s), \mintinline{sql}{SUM} (treats \mintinline{sql}{NULL}s as 0s).\\
The \mintinline{sql}{GROUP BY} clause is executed after the \mintinline{sql}{WHERE} clause, which filters the rows which are to be grouped. You can similarly filter the grouped rows with a \mintinline{sql}{HAVING} clause, of which the constraints are written the same way as \mintinline{sql}{WHERE} clauses; they're just applied to the grouped rows after the grouping rather instead of before.

\begin{minted}{sql}
SELECT group_by_column, AGG_FUNC(column_expression) AS aggregate_result_alias, ...
FROM table
WHERE condition
GROUP BY group_by_column
HAVING group_condition;
\end{minted}

You can use \mintinline{sql}{DISTINCT} when performing an aggregation, e.g. \mintinline{sql}{COUNT(DISTINCT month)} counts the unique values in the \mintinline{sql}{month} column and would return 12. It's worth noting that using \mintinline{sql}{DISTINCT}, particularly in aggregations, can slow your queries down quite a bit.

Every column specified in the \mintinline{sql}{SELECT} clause must appear in \mintinline{sql}{GROUP BY}, except columns which appear inside aggregate functions. This makes sense, because SQL wouldn't know how to combine values within each group that are not grouped on but which are \mintinline{sql}{SELECT}ed, or which of those values within each groups to show.

\subsection{\mintinline{sql}{UNION}}

When working with multiple tables, the \mintinline{sql}{UNION (ALL)} operator allows you to append the rows of one query to those of another assuming that they have the same column count, order and data type. In other words, where \mintinline{sql}{JOIN}s allow you to combine two tables side-by-side, \mintinline{sql}{UNION}s allows you to stack one table on top of the other. Without the \mintinline{sql}{ALL}, duplicate rows between the tables will be removed from the result. The \mintinline{sql}{INTERSECT} operator will ensure that only rows that are identical in both result sets are returned, and the \mintinline{sql}{EXCEPT} operator will ensure that only rows in the first result set that aren't in the second are returned (i.e. it's query order-sensitive, like \mintinline{sql}{LEFT JOIN} and \mintinline{sql}{RIGHT JOIN}). Both \mintinline{sql}{INTERSECT} and \mintinline{sql}{EXCEPT} also discard duplicate rows after their respective operations, though some databases also support \mintinline{sql}{INTERSECT/EXCEPT ALL}. These operators are rarely used in practice.

\begin{minted}{sql}
SELECT column, another_column
FROM table

UNION (ALL)/INTERSECT/EXCEPT

SELECT other_column, yet_another_column
FROM another_table
\end{minted}

\subsection{Order of execution}

Combining all the above ingredients in one query, we get the following blueprint, in which we denote the order of execution:

\begin{minted}{sql}
SELECT DISTINCT column, AGG_FUNC(column_or_expression), ... -- 5: expressions here are only executed at step 5
FROM mytable -- 1a: first executed to determine the total working set of data that is being queried
    JOIN another_table -- 1b: first executed to determine the total working set of data being queried
    ON mytable.column = another_table.column
WHERE constraint_expression -- 2: rows are filtered according to the applied constraints. Aliases in the SELECT part of the query are not yet accessible (in most RDBMSs); table aliases from FROM are available
GROUP BY column -- 3: the reminaing rows are grouped based on common values in the specified column
HAVING constraint_expression -- 4: groups are filtered according to the applied constraints. Aliases are still not yet accessible
-- 6: of the remaining rows, rows with duplicate values in the column marked as DISTINCT will be discarded
UNION (ALL)/INTERSECT/EXCEPT SELECT other_column, yet_another_column FROM another_table -- 7: append results
ORDER BY column ASC/DESC -- 8: rows are then sorted by the specified data. You can reference SELECT aliases in this clause
LIMIT count OFFSET COUNT; -- 9: rows that fall outside the range are discarded
\end{minted}

\subsection{\mintinline{sql}{CASE}}

SQL offers if-then-else functionality with \mintinline{sql}{CASE} expressions, which can occur in \mintinline{sql}{SELECT}, \mintinline{sql}{GROUP BY} and \mintinline{sql}{ORDER BY} clauses:

\begin{minted}{sql}
SELECT OrderID, Quantity,
(CASE
    WHEN Quantity > 30 THEN 'The quantity is greater than 30'
    WHEN Quantity = 30 THEN 'The quantity is 30'
    ELSE 'The quantity is under 30'
END AS QuantityText)
FROM OrderDetails
ORDER BY
(CASE
    WHEN City IS NULL THEN Country
    ELSE City
END);
\end{minted}

\subsection{Subqueries}

You can use SQL \emph{subqueries} for additional pre or post processing. A subquery must be fully enclosed in parentheses and can be referenced anywhere a normal table can be referenced: inside a \mintinline{sql}{FROM} clause, as a \mintinline{sql}{JOIN}, inside a \mintinline{sql}{WHERE} or \mintinline{sql}{HAVING} condition. When used in a condition, the subquery generally returns a one-cell result, except when using an \mintinline{sql}{IN} condition. You can alias subqueries to reference their results. You can test expressions against the results of the subquery, and you can use subqueries in expressions in the \mintinline{sql}{SELECT} clause, which allow you to return data directly from the subquery. They are generally executed in the same logical order as the part of the query that they appear in.

\begin{minted}{sql}
SELECT *
FROM sales_associates
WHERE salary > 
   (SELECT AVG(revenue_generated)
    FROM sales_associates);
\end{minted}

\emph{Correlated subqueries} reference and are dependent on a column or alias from the outer query. Unlike regular subqueries, each of these inner queries need to be run for each of the rows in the outer query, since the inner query is dependent on the current outer query row:

\begin{minted}{sql}
SELECT *
FROM employees
WHERE salary > 
   (SELECT AVG(revenue_generated)
    FROM employees AS dept_employees
    WHERE dept_employees.department = employees.department); -- The subquery needs to know what department each employee is in
\end{minted}

\section{Data manipulation language}

A \emph{data manipulation language} (DML) is for adding (inserting), deleting, and modifying (updating) data in a database. Querying is sometimes also considered a part of a DML.

\subsection{\mintinline{sql}{INSERT}}

You can insert new rows into a table with \mintinline{sql}{INSERT}, which declares which table to write into, the columns of data that we are filling, and one or more rows of data to insert. The number of values need to match the number of columns specified. If you don't specify any columns, each new row of data you insert should contain values for every corresponding column in the table. you can use mathematical and string expressions with the values that you are inserting:

\begin{minted}{sql}
INSERT INTO table (column1, column2) -- Specifying columns is optional
VALUES (row1_value_or_expr1, row1_value_or_expr2),
        (row2_value_or_expr1, row2_value_or_expr2 / 10); -- Can be any number of rows
\end{minted}

\subsection{\mintinline{sql}{UPDATE}}

\mintinline{sql}{UPDATE} takes multiple column-value pairs and applies those changes to each and every row that satisfies the constraint in the WHERE clause. If you leave out the \mintinline{sql}{WHERE} clause, you will update all rows.

\begin{minted}{sql}
UPDATE table
SET column = value_or_expr, other_column = another_value_or_expr
WHERE condition;
\end{minted}

\subsection{\mintinline{sql}{DELETE}}

\mintinline{sql}{DELETE} is similar to \mintinline{sql}{UPDATE}, but doesn't need \mintinline{sql}{SET}. Omitting the \mintinline{sql}{WHERE} clears the whole table.

\begin{minted}{sql}
DELETE FROM table
WHERE condition;
\end{minted}

\section{Data definition language}

A \emph{data definition language} (DDL) is a syntax for creating and modifying database objects such as tables, indices, and users. A \emph{database schema} (but colloquially also referred to simply as a 'schema') is the structure/blueprint of a database; the tables, their columns/attributes and associated constraints and data types, and the relationships between tables. This kind of schema is typically defined in a \mintinline{sql}{.sql} or \mintinline{sql}{.ddl} file containing \mintinline{sql}{CREATE TABLE} statements. Another example of a DDL is XML.

\subsection{\mintinline{sql}{CREATE TABLE}}

You can create a new table with \mintinline{sql}{CREATE TABLE}. Use \mintinline{sql}{IF NOT EXISTS} to suppress errors and skip creating the table if one with the same name already exists. Every line inside the statement defines a column (or a table-wide constraint).

\begin{minted}{sql}
CREATE TABLE [IF NOT EXISTS] table (
    <column1> <datatype> <column_constraint>,
    <column2> <datatype> <column_constraint>
    [, <table_contraint>]
);
\end{minted}

A \mintinline{sql}{<datatype>} can be any of \mintinline{sql}{INTEGER}, \mintinline{sql}{BOOLEAN}, \mintinline{sql}{FLOAT}, \mintinline{sql}{DOUBLE}, \mintinline{sql}{REAL}, \mintinline{sql}{TEXT}, \mintinline{sql}{CHARACTER(max_num_chars)}, \mintinline{sql}{VARCHAR(max_num_chars)}, \mintinline{sql}{DATE}, \mintinline{sql}{DATETIME}, \mintinline{sql}{DEFAULT <default_value>}, \mintinline{sql}{BLOB} (Binary Large OBject, often opaque to the database, so you usually have to store them with the right metadata to requery them).

Constraints can be column-specific (in which case they're defined on the corresponding line of the column) or apply to the whole table (in which case they're defined on a separate line. Column constraints can refer to only one column, table constraints can refer to multiple columns. With the exception of \mintinline{sql}{NOT NULL}, both types of constraint can be any of:

\begin{itemize}
    \item \mintinline{sql}{PRIMARY KEY}; the values in this column are unique, and each value can be used to identify a single row in this table. These are implicitly \mintinline{sql}{NOT NULL} and \mintinline{sql}{UNIQUE}. You can only have one primary key, but you can have multiple columns in your primary key by declaring it as a table constraint like \mintinline{sql}{PRIMARY KEY(<some_column>, <another_column)}.
    \item \mintinline{sql}{AUTOINCREMENT}; integer values are automatically filled in and incremented with each row insertion. Not supported in all databases.
    \item \mintinline{sql}{UNIQUE}; the values in this column have to be unique, so you can't insert another row with the same value in this column as another row in the table.
    \item \mintinline{sql}{NOT NULL}; inserted values can't be \mintinline{sql}{NULL}. If they can be \mintinline{sql}{NULL}, you can optionally specify the \mintinline{sql}{NULL} column constraint to make this explicit.
    \item \mintinline{sql}{CHECK(<expression>)}; checks the expression on inserted rows for validity. You can also put a \mintinline{sql}{CHECK} on a separate row to enforce a condition on an entire record.
    \item \mintinline{sql}{FOREIGN KEY}; a consistency check which ensures that each value in this column corresponds to another value in a column in another table.
\end{itemize}

You can optionally name constraints with the \mintinline{sql}{CONTRAINT <contraint_name>} keyword right before the constraint definition. Whenever you want to remove a constraint with the \mintinline{sql}{ALTER TABLE} command, you specify it with this name. If a name isn't supplied, the DBMS will automatically generate a constraint name, but you will have to look up its generated name in the database's metadata first if you ever want to remove it.

\subsection{\mintinline{sql}{ALTER TABLE}}

You can update an existing table's schema with \mintinline{sql}{ALTER TABLE}, which supports a number of operations. You can add columns:

\begin{minted}{sql}
ALTER TABLE table
ADD <another_column> <datatype> [column_constraint];
\end{minted}

Default value constraints are applied to existing and new rows.\\
In some databases like MySQL, you can specify where to insert the new column using the \mintinline{sql}{FIRST} or \mintinline{sql}{AFTER} clauses, though this is not standard.

You can drop a constraint:

\begin{minted}{sql}
ALTER TABLE table
DROP CONSTRAINT <constraint_name>; 
\end{minted}

You can drop columns:

\begin{minted}{sql}
ALTER TABLE table
DROP column_to_be_deleted;
\end{minted}

Some databases (including SQLite) don't support this feature, so you may have to create a new table and migrate the data over.

You can rename tables:

\begin{minted}{sql}
ALTER TABLE table
RENAME TO new_table_name;
\end{minted}

\subsection{\mintinline{sql}{DROP TABLE}}

You delete a table with:

\begin{minted}{sql}
DROP TABLE [IF EXISTS] table;
\end{minted}

If you have another table that is dependent on columns in the table you are removing, then you will have to either remove the dependent rows from those tables or remove them entirely.

\subsection{\mintinline{sql}{CREATE SCHEMA}}

Besides being used in a broader design sense to refer to the structure of a database's tables, the word 'schema' is also confusingly used to refer to the concept of a namespace in a database. MySQL uses the word 'schema' interchangeably with 'database', such that \mintinline{sql}{CREATE SCHEMA} and \mintinline{sql}{CREATE DATABASE} are aliases for the same command. In SQL Server and PostgreSQL, \mintinline{sql}{CREATE DATABASE} really does create a database and \mintinline{sql}{CREATE SCHEMA} creates a new schema entity in the current database.\\
In PostgreSQL a database contains one or more named schemas, which in turn contain tables and other kinds of named objects (e.g. data types, functions, views, operators, etc.). A schema can be considered a namespace; within one schema, two objects of the same type cannot have the same name. Whereas a client connection to a server can only access data in a single database (the one specified in the connection request), schemas are not rigidly separated, and therefore a user can access objects in any of the schemas in the database they are connected to (if they have the priviliges). You can use schemas to allow many users to use one database without interfering with each other, to organize database objects into logical groups to make them more manageable, or for putting third-party applications into separate schemas so they do not collide with the names of other objects. Schemas are analogous to directories at the operating system level, except that schemas cannot be nested.

\begin{minted}{sql}
CREATE SCHEMA some_schema;
\end{minted}

You can refer to tables as \mintinline{sql}{schema.table}, or, even more generally, as \mintinline{sql}{database.schema.table}.

\section{Data control language}

A \emph{data control language} (DCL) is used to control access to data stored in a database (authorization).

\subsection{\mintinline{sql}{GRANT}}

Gives specified permissions for the table (and other objects) to, or assigns a specified role with certain permissions to, specified groups or users of a database.

\begin{minted}{sql}
GRANT SELECT, INSERT, UPDATE, DELETE on Employees TO User1
\end{minted}

\subsection{\mintinline{sql}{REVOKE}}

Takes away specified permissions for the table (and other objects) to, or takes away a specified role with certain permissions to, specified groups or users of a database.

\begin{minted}{sql}
REVOKE INSERT On Employees TO User1
\end{minted}

\subsection{\mintinline{sql}{DENY}}

Denies a specified permission to a security object.

\begin{minted}{sql}
DENY UPDATE On Employees TO User1
\end{minted}

\section{Stored procedures}

A \emph{stored procedure} is a precompiled SQL statement that you can store in the database data dictionary (tables with metadata), so the code can be reused and executed as a single unit. Of course you can have such functionality/code outside of the database in your application, but those need to transfer the data required to perform a task or calculation, whereas a stored procedure works with the data directly and is often much faster than transferring data back and forth. On the other hand, stored procedures add processing load to the database, which can already be a choke point for performance, and you should consider whether you want to have another location with functional code.

In PostgreSQL\footnote{Also here, syntax may differ between database systems.}, you create a stored procedure as follows:

\begin{minted}{sql}
CREATE PROCEDURE <procedure>(<parameter1> data_type>, ...)
AS
BEGIN
    SELECT CustomerName
    FROM Customers;
END
\end{minted}

and you call them using \mintinline{sql}{CALL <procedure>(<parameter1>, ...)} (or \mintinline{sql}{EXECUTE} or \mintinline{sql}{EXEC}). Names of stored procedures are conventionally prefixed with \mintinline{sql}{sp_}.
Stored procedures are similar to prepared statements, but those are simpler and more declarative, prepared statements are not ordinarily written to use procedural logic and cannot operate on variables. Because of their simple interface and client-side implementations, prepared statements are more widely reusable between DBMSs.
