\subsection{Memory}

Data is stored in RAM, which is a sequential series of bytes, each of which has an address. It stands for Random Access Memory, because you can access any position/address in RAM in constant time. If you want to store any data, you have to ask your OS, which will \emph{allocate} (reserve) the requested amount of contiguous space and return the address of the first byte.

\subsection{Arrays}

Arrays are stored contiguously in RAM.

\emph{Static arrays} have a fixed length. It never needs to be moved when it runs out of space, but it is highly inflexible. Accessing the $i$'th element is $O(1)$ (\emph{random access}), inserting and/or removing the last element is $O(1)$ (which is just overwriting it), inserting and/or removing anywhere else is $O(n)$, because you need to shift every element. For this reason, writing to an array in practice is often only overwriting the value at a certain index.

\emph{Dynamic arrays} double in size whenever they run out of space, with the new array allocated somewhere else. All time complexities are still the same. Even pushing $n$ elements to an initially empty array is still only $O(n)$, because it takes $1 + 2 + 4 + 8 + ... + \frac{n}{2} + n$ operations, which is never more than $2n$.

\subsection{Linked lists}

A \emph{(singly) linked list} consists of nodes; objects containing a value and a pointer (address) to the next node. Linked lists do not need to be stored contiguously and never need to move. Accessing and searching is $O(n)$ due to traversal (\emph{sequential access}), except at the start or end when maintaining a \mintinline{java}{head} or \mintinline{java}{tail} pointer. Inserting/deleting is $O(1)$, because you only have to change some pointers (not taking into account the lookup/access of the location). Note that the time complexities are basically the inverse of an array: finding the point of insertion/deletion is $O(n)$ for linked lists and $O(1)$ for arrays, while performing the insertion/deletion is $O(1)$ for linked lists and $O(n)$ for arrays. In general, arrays are better for reads (and overwrites), linked lists are better for writes.

A \emph{doubly linked list} has nodes which also point to the previous node. Deleting at the end is now also $O(1)$, because we have a pointer to the penultimate node (\mintinline{java}{tail.prev}). Java has \mintinline{java}{LinkedList<T>}, a doubly-linked list implementation of the \mintinline{java}{List} and \mintinline{java}{Deque} interfaces.

\subsection{Stacks, queues, and deques}

A \emph{stack} is a LIFO (last in, first out) data structure which supports pushing, popping and peeking (at the top) in constant time. Stacks can be implemented as dynamic arrays or linked lists.

A \emph{queue} is a FIFO (first in, first out) data structure which supports enqueueing to the start and dequeueing from the end in constant time. Are best implemented as linked lists.

A \emph{double-ended queue} is like a generalization of a stack and a queue which supports $O(1)$ pushing and popping to either end. Python has \mintinline{python}{deque} in the \mintinline{java}{collections} module. Java has the \mintinline{java}{ArrayDeque} and \mintinline{java}{LinkedList} container classes that both implement interface \mintinline{java}{Deque}, which you can use as a stack with \mintinline{java}{push()}/\mintinline{java}{addFirst()} and \mintinline{java}{pop()}/\mintinline{java}{removeFirst()}/\mintinline{java}{remove()} for adding/removing on top/at the front, or as a queue with \mintinline{java}{add()}/\mintinline{java}{addLast()} (at the end) and \mintinline{java}{pop()}/\mintinline{java}{removeFirst()} (at the front).

\subsection{Hash tables}

A \emph{hash function} is a function where you put in a string or any kind of sequence of bytes, and you get back a number. The function has to be consistent (same input maps to the same output). This output number can be used to deterministically index into an array by modulo'ing it by the array size. A \emph{hash table}, \emph{(hash) map}, \emph{dictionary} (dicts in Python), or \emph{associative array} is a hash function with an array. Java has \mintinline{java}{HashSet}/\mintinline{java}{HashMap}.

\emph{Collisions} happen when different keys map to the same index. Collisions are avoided with a good hash function -- one that distributes values in the array evenly -- and a low load factor. The \emph{load factor} of a hash table is the percentage of occupied slots in the array. Having a load factor greater than 1 means you have more items than slots in your array. Each time the load factor is above something like $0.7$, the array is doubled\footnote{The size of the array should actually be a prime number to further avoid collisions for math reasons, so in practice it's not quite doubling it when resizing.} and all existing key-value pairs are moved to their new hashing location (which is dependent on array size), and this is called "rehashing the array". But collisions may still occur, and they can be handled in different ways:

\begin{itemize}
    \item \emph{Chaining}: instead of one key-value pair per index, store a linked list per index.
    \item \emph{Open addressing}: if you want to store a key-value pair and an index is occupied, you just put the key-value pair at the next unoccupied index. When searching, you search (\emph{probe}) until the next unoccupied index (at which point you know it's not in there, or else it would be at that spot). Note that you need to store the key with every value for this to work, hence storing key-value pairs. The downside of +1'ing (linear probing) is that you cluster all elements, and there are better ways, like continuously squaring and modulo'ing (quadratic probing).
\end{itemize}

Hash tables have $O(1)$ time complexity for inserting, removing and searching.\footnote{Actually, $O(1)$ is the average time complexity and $O(n)$ is the worst time complexity if you have a bad hash function, but you can basically assume that $O(1)$ is simply the worst case.} The caveat is that you can't easily traverse it in order, which would take $O(n \log n)$. They also can't contain duplicates. Use cases are fast lookup, filtering out duplicates, and caches.

\subsection{Graphs}

\emph{Graphs} have $n$ nodes/vertices and $m$ edges, where the edges may have a direction (represented with an arrow rather than a line). For directed graphs, $m \leq n^2$ (if vertices can point to themselves). In undirected graphs, edges don't have a direction. In directed graphs, nodes have in-neighbours (that points toward them) and out-neighbours.

Graphs are most often represented as an \emph{adjacency list}, which concretely is either

\begin{itemize}
    \item a collection of node/vertex objects, each of which has a value and a list of references to other node objects.
    \item a hash table with the node value as key and a list of (out-)neighbours as values (in which each element is that neighbour's node value). Each node must have a unique value for this to work.
    \item a list of node pairs.
\end{itemize}

A \emph{topological sort} is (one of potentially multiple possible) valid ordered lists of a directional graph. You can use DFS or Kahn's algorithm (see Section \ref{kahns_algorithm}) to get a topological sort (both are also useful for cycle detection).

In a \emph{weighted graph}, each edge has an associated \emph{weight}. One way of implementing/representing a weighted graph in code is with a nested hash table: the top-level table maps each node to a nested table, which in turn maps (only) that node's out-neighbours to their weights.

Graphs are less commonly represented as a matrix; a 2D array in which each \mintinline{java}{0} is a node and each \mintinline{java}{1} is not, and then you can move up, down, left, right (and sometimes diagonally) to each neighbouring node. So the graph is laid out like a grid, with possible holes. The least common graph representation is the \textit{adjacency matrix}, a 2D array in which each cell represents an edge from node $i$ to node $j$, and the value indicates whether the edge exists. Takes $O(V^2)$ space. Uncommon on interviews.

\subsubsection{Trees}

A \emph{tree} is a graph that is both \emph{connected} (there is path between every pair of nodes) and \emph{acyclic} (there are no cycles). \emph{Rooted trees} have exactly one node that leads to all the other nodes: the \emph{root}, which doesn't have parents. Nodes can have children and a parent. Nodes with no children are \emph{leaf nodes}. A valid tree has exactly $m = n-1$ edges.

A \emph{binary tree} is a special type of tree where nodes can have at most two children. They're highly similar to linked lists, but with different terminology and no cycles. Rather than one \mintinline{java}{next} pointer, each node has two \mintinline{java}{left} and \mintinline{java}{right} pointers, and likely also a \mintinline{java}{parent} pointer. The \emph{height} of a binary tree is the longest path from the root node to any leaf node. The \emph{depth} of a node is the length of the path to the root. Huffman coding (the foundation for text compression algorithms) uses binary trees.

\paragraph{Binary search trees}

In a \emph{binary search tree} (BST), everything in a node's left/right subtree is less/greater than the node's value. A balanced BST is one where each node's left and right subtree have equal height, both $\log(n)$. Searching a balanced tree is $O(\log n)$, searching a super unbalanced tree (basically a linked list) is $O(n)$, and generally it is $O(h)$, where $h$ is the height. Finding the minimum/maximimum is easy: just keep taking left/right turns, which is still $O(\log n)$. The main advantage of BSTs over (sorted) arrays is that adding/removing elements/nodes can also be $O(\log n)$ (whereas it is $O(n)$ for arrays). It's somewhat of a middle ground between arrays and linked lists. They're slower than hash sets/tables ($\Theta(\log n)$ vs. $\Theta(1)$), but you can traverse them in order, whereas this isn't possible with hash tables.

Adding nodes as leaf nodes is easier than adding nodes as root or parent nodes, but does result in unbalances trees. This way is very similar to searching; we essentially search for the value to insert until we reach a null, and then create and add the node there.\\
Removing a node is more involved. First we need to search for it. If the node has no children, just remove it and set the parent's pointer to null. If the node has one child, set the parent's pointer to point to that child, bypassing the node. If the node has two children, we take the leftmost descendant (with the smallest value) from the right subtree, and replace the node with that. You can also take the rightmost descendant (with the largest value) from the left subtree.

Sets and maps can be implemented with BSTs. Java has \mintinline{java}{TreeSet}/\mintinline{java}{TreeMap} (which implement \mintinline{java}{SortedSet}/\mintinline{java}{SortedMap}). Python calls BST maps \mintinline{python}{SortedDict} (you need package \mintinline{python}{sortedcontainers}). C++ uses BSTs for \mintinline{cpp}{map<T,T>}. 

\paragraph{AVL trees}

\emph{AVL trees} are a type of self-balancing BST, i.e. it will rebalance whenever the tree is out of balance (with a height difference  between leaf nodes of more than 1) by rotating. A \emph{rotation} is an operation on a binary tree that changes the structure without interfering with the order of the elements. It moves one node up in the tree and one node down. Lookup, insertion, and deletion all take $O(\log n)$ time in both average and worst case. Each node stores either its height or its \emph{balance factor}, which tells which child subtree is taller and by how much. Negative values indicate the left tree is taller and vice versa, and a $0$ indicates a balanced tree. $-1$ or $1$ are okay, because AVL trees allow a maximum difference of $1$. If you have the heights of each subtree, it is easy to compute the balance factor by subtracting the right subtree's high by that of the left subtree. New nodes are added as leaf nodes (with height or balance factor $0$), and all its ancestors heights/balance factors are updated. If a node's balance factor becomes $-2$ or $2$, it is immediately rebalanced. After a rebalance, you don't need to move further up the tree, because AVL trees require one rebalancing at most.

\paragraph{Splay trees}

A different take on balanced BSTs. If you have recently looked up an item, the next lookup for that item will be faster, because when you look up a node, it will make that node the new root. More generally, recently looked up nodes get clustered to the top and become faster to look up.

\paragraph{B-trees}

B-trees are a generalized form of binary tree, where each node can have more than one key and more than two children. More specifically, a node's number of children is one greater than it's number of keys. \emph{Seek time} is the time it takes for the hard disc's read/write head to move to the correct track, i.e. it's a physical mechanism that introduces a delay. B-trees optimize this seek time. They spend more time reading each node, but they seek less because they read more data into memory in one go. For this reason, they're a popular data structure for databases. Like BSTs, everything in a node's left/right subtree is less/greater than the node's value.

\paragraph{Tries}

A \emph{trie}, or prefix tree, is a rooted tree data structure used for retrieval of a key in a dataset of strings. It's used for autocomplete, spellchecking, IP routing, etc. Each node in the tree has a link for each possible character (which by default are null) and a boolean indicating if the current node ends a valid word that has been added (else its just a prefix).\footnote{You can also visualize it as the nodes representing characters, with the root node representing the empty string "", but I find it more intuitive to have the links contain the characters.} If $m$ is the word/key length, insertion is $O(m)$ time and space, searching for the exact input word and searching any word with the given prefix is $O(m)$ time and $O(1)$ space. For insertion and search, hash tables are equally as good, but they don't support prefix search.

\paragraph{Segment trees}

A \emph{prefix}/\emph{postfix} is a subarray (contiguous) that starts at the beginning/ends at the end of an array. Prefix sums allow quickly querying the sum of a subarray: if you have the prefix sum up until each index of an array, you can query the sum/product of any subarray demarcated by \mintinline{python}{i,j} in constant time as \mintinline{python}{prefixSum[j] - prefixSum[i-1]}. (This also works for (prefix) products.) However, you can't update elements. Segment trees allow for both querying subarray sums and updating elements, both in $O(\log n)$. Segment trees are balanced binary trees, where each node stores the left and right (inclusive) indices of the subarray it represents, the sum of that subarray, and pointers to its left and right children, where the left child represents range \mintinline{python}{L,M} and the right child represents \mintinline{python}{M+1,R}. The tree is built bottom-up, so the root node takes \mintinline{python}{0, arr.length-1} and recursively splits that at \mintinline{python}{M = (L+R)//2} and creates the child nodes, and after both recursions, computes its sum from the sums of its children. The base case is if \mintinline{python}{L==R}, which is a leaf node representing one element.
Updating an index \mintinline{python}{i} entails recursively traversing the tree, at each node picking the left child if \mintinline{python}{i <= (L+R)//2} and the right child if \mintinline{python}{i > (L+R)//2}, and at the base case/leaf node, updating the sum and recalculating all parent sums, just as when building the tree.
Searching the sum of subarray \mintinline{python}{L,R} entails recursively traversing the tree as well. At each node, if \mintinline{python}{l == L && r == R}, we return the node's sum, else if \mintinline{python}{l > (L+R)//2}, we only recurse right, if \mintinline{python}{r <= (L+R)//2}, we only recurse left, and else we need to recurse both and add the results. Often, the leaf nodes aren't even reached, but we can backtrack earlier already.

\subsubsection{Forest}

A \emph{forest} is a disjoint union (basically a collection) of trees.

\paragraph{Union-find} \label{union_find}

A \emph{union-find}, also known as a disjoint-set or merge-find, stores a collection of disjoint (non-overlapping) sets -- a partitioning of a set into disjoint subsets. They're often implemented as a forest of trees data structure that can be used on undirected graphs with disjoint connected components for counting the number of connected components and for cycle detection. The trees aren't meant to accurately represent the graph. Given the number of nodes and a list of unique, undirected edges (so \mintinline{java}{[1,2]} and \mintinline{java}{[2,1]} won't both appear), we assume each node is disjoint initially. We keep track of a parent pointer for each node (stored in a hash map, mapping node value to parent node), and the rank (height) of each tree (also stored in a hash map, with the root values as keys), which we want as small as possible for best performance. Iterating through the list of edges, for each edge (i.e. pair of nodes), we find the roots of the trees of both nodes by following the nodes' parent pointers. If the roots are the same, the nodes are already in the same tree, and since edges in the list are unique, there is a cycle. Else, we parent the root of the smallest tree (\mintinline{java}{ranks[root]}) to the root of the largest tree. If both trees have the same rank, choose one of two parentings and increase the relevant rank. While finding the roots, we can do path compression, i.e. at all levels/calls of recursion, parent the result of the subrecursion (the found root) to the input node.

\subsection{Priority queues}

Priority queues are like queues, except the order of popping is done on the minimum/maximum value. Lookup of this value is $O(1)$ (compared to $O(\log n)$ for BSTs), because that element is always the root. Popping it and replacing it with the next value is $O(\log n)$ time. Adding a value is $O(\log n)$. They're often implemented with heaps, but implementations with other data structures are possible. \emph{Heaps} are binary trees with two properties:

\begin{itemize}
    \item The \emph{structure property} dictates that it is a complete binary tree, meaning only the lowest level may have holes, and those holes should be on the right (so nodes are added in breadth-first order).
    \item The \emph{order property} states that the value of a node should be smaller/larger than or equal to all values in both its subtrees.
\end{itemize}

Duplicate values are allowed. Under the hood, heaps use arrays. The $0$'th index is skipped, and all other nodes are in the array in breadth-first order. For every node at index $i$, its left child is at index $2i$, its right child at index $2i+1$, and its parent at $\lfloor \frac{i}{2}\rfloor$.

Adding a node is done by putting it at the next empty spot, and then keep swapping it with its (current) parent while it is larger/smaller than that parent (a good term for it is "percolating" upwards). Popping the highest priority node is done by returning and removing the root node, replacing it with the last node, and then percolating that down, i.e. continually replacing it with the smallest/largest of its two children.

You can heapify an existing array in $O(n)$ time by first moving the first element to last to create space for the 0'th dummy element and then looping from the back of the array to the front, percolating each element down.\footnote{You can actually start the algorithm at index $\lfloor \frac{i}{2}\rfloor$, because that's the last node in the array with at least one child.} Because $\frac{n}{2}$ nodes don't move at all, $\frac{n}{2}$ only move one down, $\frac{n}{4}$ move at most two down, etc., you get a time series that basically approximates $n$.
Searching a priority queue for an element is $O(n)$, but that's not their use anyways. Java has the (heap-based) \mintinline{java}{PriorityQueue}, which implements \mintinline{java}{Queue} and supports \mintinline{java}{add(E e)}, \mintinline{java}{peek()} and \mintinline{java}{poll()} (which pops the head).

\section{Algorithms}

\subsection{Big O notation}

\emph{Big O notation} tells you how fast an algorithm is and how the running time or memory allocation size increases as the input size increases, assuming worst case scenarios. It is expressed as the number of operations or units of memory as a function of the input size. Constants are left out, because those capture hardware/performance/implementation differences that are irrelevant and are therefore abstracted away. Common complexities are, from fastest to slowest:

\begin{itemize}
    \item $O(1)$, or \emph{constant time}, e.g. hash table lookup.
    \item $O(\log n)$, or \emph{logarithmic time}, e.g. binary search.
    \item $O(n)$, or \emph{linear time}, e.g. linear search.
    \item $O(n \log n)$, or \emph{linearithmic time}, e.g. quicksort.\footnote{Exponentiation is another example of a calculation that costs $O(\log n)$, though some languages may be implemented in a way that exponentiation on integers costs $O(1)$.}
    \item $O(n^2)$, or \emph{quadratic time}, e.g. selection sort.\footnote{If you come up with an algorithm that is $O(n^2)$, always consider if sorting the input array ($O(n \log n)$) allows solving the problem in $O(n \log n)$ or better.}
    \item $O(n^3)$, or \emph{cubic time}.
    \item $O(n^k)$, or \emph{polynomial time}, where $k$ is a constant. A generalization of quadratic and cubic time.
    \item $O(k^n)$, or \emph{exponential time}.
    \item $O(n!)$, or \emph{factorial time}, e.g. traveling salesman.
\end{itemize}

\subsection{Recursion}

\emph{Recursion} happens when a function calls itself. There’s no performance benefit; it is used when it makes the solution clearer. Every recursive function has a \emph{base case} (or else it would loop infinitely) and the \emph{recursive case}. Each of the recursive function calls takes up some memory on the \emph{call stack}. If your recursion may go too deep and the call stack runs out of space, you can rewrite your code to use a loop instead, or use \emph{tail recursion}.

One-branch recursion (for example: calculating a factorial) is at least $O(n)$ time and space complexity (because the function call takes up space). Two-branch recursion (for example: calculating Fibonacci sequence) is also at least $O(n)$ time and space complexity.

\subsection{Dynamic programming} \label{dynamic_programming}

\emph{Dynamic programming} (DP) breaks a problem down into subproblems and solves each of those while storing the results. Can replace brute-force recursion when there is a lot of repeated work being done. For a problem to be solved by dynamic programming, it needs to have:

\begin{itemize}
    \item \emph{Optimal substructure}: the optimal solution can be obtained by the combination of optimal solutions to its sub-problems. The subproblems are usually described by means of recursion.
    \item \emph{Overlapping sub-problems}: the solutions to the subproblems are needed again and again in computing any of the superproblems. You could recompute them each time, but this often leads to exponential complexity, hence dynamic programming algorithms store them for lookup later, called \emph{memoization}.
\end{itemize}

If a problem has optimal substructure, but non-overlapping sub-problems, you'd need a divide-and-conquer algorithm instead.

Dynamic programming solutions consist of three components: a data structure to store answers to subproblems (typically a 1D or 2D array or hash map, but can also be a small, fixed number of variables), a recurrence relation describing how an answer depends on answers to subproblems, and a base case.

There are two variations, and in principle both are always possible solutions:

\begin{itemize}
    \item Top-down DP still uses recursion, but with caching added. It starts with the main problem and works down.
    \item Bottom-up DP is iterative and starts with base cases and then works towards the main problem. Often called "true DP", and likely has better space complexity than top-down. In 1D dynamic programming, if the answer to subproblem $i$ only depends on only a small constant of preceding answers $i-1, i-2, ...$, you can save space by only memorizing those last couple of answers, rotating them in a constant-size array. In some cases, the answer of a subproblem depends on all preceding answers, in which case you'd get $O(n^2)$ time complexity, but it is still dynamic programming.
\end{itemize}

Giveaways that a problem should be solved with dynamic programming are: asking for the minimum or maximum of something, having to make decisions that may depend on previously made decisions (very typical of a problem involving subsequences), or when you’re trying to optimize something given a constraint.

\subsection{Divide-and-conquer}

\emph{Divide-and-conquer} algorithms are multiple-branch recursive algorithms that divides or splits up a problem in smaller problems, each is solved recursively, until a base case is hit. Problems that can be solved with divide-and-conquer exhibit optimal substructure, but not overlapping sub-problems (see Section \ref{dynamic_programming}), so there's no point in storing solutions to sub-problems.

\subsection{Binary search} \label{binary_search}

Binary search of an element in a sorted array is a divide-and-conquer algorithm with $O(\log n)$ time complexity and $O(1)$ space complexity.

\begin{minted}{python}
def binary_search(nums, target):
    if len(nums) == 0:
    	return -1
    
    lo, hi = 0, len(nums) - 1
    while lo <= hi:
    	mid = (lo + hi) // 2
    	if nums[mid] > target:
    		hi = mid - 1
    	elif nums[mid] < target:
    		lo = mid + 1
    	else:
    		return mid
    
    return -1
\end{minted}

Problems that require minimizing the maximum or maximizing the minimum often suggest a binary search approach on the range of possible values for this maximum or minimum, respectively.

\subsubsection{Selection sort}

The simplest sorting algorithm is \emph{selection sort}, which goes through the list searching for the minimum element again and again. $O(n^2)$ time complexity, $O(1)$ space complexity.

\subsubsection{Insertion sort}

\emph{Insertion sort} is stable and $O(n^2)$. Outer loop loops with \mintinline{java}{i} through the array from front + 1 to back. Everything to the left of \mintinline{java}{i} is sorted. Inner loop brings the element at \mintinline{java}{i} from \mintinline{java}{i} backwards to its place by swapping as long as it's out of order.

\subsubsection{Mergesort}

\emph{Mergesort} is divide-and-conquer (two-branch recursion); it splits the array into two newly allocated arrays, recursively sorts each half and then merges those sorted subarrays into the original array. Time complexity is $O(n \log n)$: there are $\log n$ levels/splits that each require $n$ operations. It's stable if you take care to prioritize the first half when merging.

\begin{minted}{python}
def mergeSort(array):
    if len(array) < 2:
        return
    
    left = array[:len(array)//2]
    right = array[len(array)//2:]
    mergeSort(left)
    mergeSort(right)

    i = j = 0

    # Add until we reach end of either left or right
    while i < len(left) and j < len(right):
        if left[i] < right[j]:
            array[i+j] = left[i]
            i += 1
        else:
            array[i+j] = right[j]
            j += 1

    # Add remaining elements. Note that only one of these loops will execute
    while i < len(left):
        array[i+j] = left[i]
        i += 1

    while j < len(right):
        array[i+j] = right[j]
        j += 1
\end{minted}

Mergesort can theoretically be done in-place (avoiding extra array allocation), but implementing the merge step is messy and complex, and it's usually slower than the standard version because of all the extra element shifting and careful indexing.

\subsubsection{Quicksort} \label{quicksort}

Quicksort, developed by Tony Hoare, is similar to mergesort. It is also divide-and conquer, but no additional arrays are needed; everything happens in-place. Also, quicksort first partitions the array and then makes two recursive calls, whereas mergesort first makes recursive calls for the two halves and then merges them. At each level, the rightmost element is selected as pivot, and the rest of the array is looped front-to-back, and if an element is smaller than the pivot, it is swapped with the element at a separate "swapper" pointer that also starts at the beginning and is incremented with every swap. Lastly, the pivot itself is swapped with the element at the swapper pointer, and it then separates the array in two halves, each of which is unsorted but all elements on the left side are smaller than all elements on the right side, and each side is recursed (without the pivot).

\begin{minted}{python}
def quicksort(nums):
    if len(nums) < 2:
        return nums
    else:
        pivot = nums[0] # This isn't using Lomuto partition scheme, which uses the last element
        less = [i for i in nums[1:] if i <= pivot]
        greater = [i for i in nums[1:] if i > pivot]
        return quicksort(less) + [pivot] + quicksort(greater)
\end{minted}

Worst time complexity is $O(n^2)$ with very unlucky pivots, but in practice it's $O(n \log n)$. Even though quicksort is $\Theta(n \log n)$ (on average) and merge sort is $O(n \log n)$ (always), in practice, quicksort's average case is faster than mergesort's average case and quicksort hits its average case way more often than its worst case. The performance of quicksort heavily depends on the choice of pivot, but always taking a random pivot will guarantee the $\Theta(n \log n)$ average runtime.

The above method of partitioning the array into halves smaller and larger than the pivot is called Lomuto partition scheme. There are many partition schemes, but this one is compact and easy to understand, though less efficient than Hoare's original scheme, e.g. when all elements are equal. With this scheme, time complexity degrades to $O(n^2)$ when the array is already in order, due to the partition being the worst possible one.

You can’t sort an array in $O(n)$ time on a single CPU core, but there is a parallel version of quicksort that will sort an array in $O(n)$ time.

\emph{Quickselect}, also developed by Tony Hoare, is a highly similar \emph{selection algorithm}, which are algorithms for finding the $k$th smallest (or largest, by tweaking the below code slightly) element in an unordered list. The array is partitioned in two halves just like in quicksort, but because we know in which partition the desired element lies, only one side needs to be recursed. This has time complexity $\Theta(n)$ on average (which is almost always the case in practice), and $O(n^2)$ worst-case complexity, but this only happens on edge cases.

\begin{minted}{python}
def kthSmallest(nums, k):
    def partition(left, right, pivot):
        nums[pivot], nums[right] = nums[right], nums[pivot]
        
        store_index = left
        for i in range(left, right):
            if count[nums[i]] < nums[pivot]:
                unique[store_index], unique[i] = unique[i], unique[store_index]
                store_index += 1

        nums[right], nums[store_index] = nums[store_index], nums[right]
        
        return store_index
    
    def quickselect(left, right, k) -> None:
        if left == right: # Base case; the list contains only one element
            return
        
        pivot = random.randint(left, right) # Select a random pivot index
        pivot = partition(left, right, pivot) # Find the pivot index in a sorted list
        
        if k == pivot: # If the pivot is in its final sorted position
             return
        elif k < pivot:
            quickselect(left, pivot - 1, k)
        else:
            quickselect(pivot + 1, right, k)

    n = len(nums)
    quickselect(0, n - 1, k)
    return nums[k]
\end{minted}

\subsubsection{Bucket sort}

Bucket sort is $O(n)$, but is unstable and can only be used when all values of elements are from some small set. In one pass of the array, all possible values are counted in a separate array where the indices are the values, and in a second pass, the original array is overwritten using these counts. Only used in certain situations.

In a more general formulation, the buckets span ranges of values, and the buckets are individually sorted with some other sorting algorithm before being concatenated.

\subsubsection{Breadth-first search (BFS)}

Either for determining whether a path exists between two nodes, or for finding the path along the least number of edges. BFS radiates out from the starting point and is $O(n+m)$.

\begin{minted}{python}
def bfs(searched_item):
    queue = deque()
    queue += graph[searched_item]
    searched = set()

\subsubsection{Depth-first search (DFS)}

Useful for topological sort, cycle detection, and path detection. Ill-suited for finding a shortest path.

\begin{minted}{python}
def dfs(graph, start, visited=None):
    if visited is None:
        visited = set()
    if start in visited:
        return
    visited.add(start)
    for neighbor in graph.get(start, []):
        dfs(graph, neighbor, visited)
    return visited
\end{minted}

DFS can also be done iteratively with a stack, in which case the algorithm is almost identical to BFS (just with a stack instead of a queue).

\subsubsection{Dijkstra's Algorithm}

For finding the shortest path from \mintinline{java}{start} to \mintinline{java}{end} in a weighted graph. Only works on graphs with no cycles and where all the edges are nonnegative.

\begin{enumerate}
    \item Initialize: create two hash tables: one that maps each node to the sum of its shortest paths from \mintinline{java}{start} (initialized as $\infty$) and one that maps each node to its parent/preceding node on the shortest path. Set the path costs of \mintinline{java}{start}'s out-neighbours (and \mintinline{java}{start} as their parent). Create a set for processed nodes.
    \item Loop until all nodes are processed:
    \begin{enumerate}
        \item Find the node with currently the shortest path from \mintinline{java}{start}. The key idea of the algorithm is that there is no other shorter path to this node.\footnote{This is also why negative weights break the algorithm; the algorithm relies on the fact that, once you process a node, there shouldn't be a shorter path to it, but with negative weights, a shorter path can still turn up. The Bellman–Ford algorithm can deal with negative weights.}
        \item For all this node's out-neighbors, if the route via this node is shorter, update that out-neighbours shortest path cost and set it's parent to this node. Add the node to the set of processed nodes.
    \end{enumerate}
    \item Calculate the final path by following the parent nodes back to \mintinline{java}{start}.
\end{enumerate}

Assuming the graph is given as a nested graph:

costs = {}

\subsubsection{Kahn's algorithm} \label{kahns_algorithm}

Kahn algorithm is one way to get a topological sorting of a directed graph, or to determine whether it has a cycle (DFS can also do both). Starting with an empty list for the ordered nodes and a queue of leaf nodes (with no incoming edges), you iterate until the queue is empty: pop a node from the queue, add it to the list, and for every out-neighbour: remove the edge from the graph, and add the out-neighbour to the queue if it has no in-neighbours anymore. If your ordered list is the total number of nodes, there are no cycles.

\subsubsection{Trees}

\paragraph{DFS on BST}

Depth-first search on BST:

\begin{minted}{python}
def dfs(root):
    if not root:
        return    
    dfs(root.left)
    print(root.val) # Inorder traversal
    dfs(root.right)
\end{minted}

In preorder traversal, the operation on the node is done before recursing into the left subtree. In postorder traversal, the operation is done after recursing into the right subtree.

\begin{figure}[H]
    \centering
    \includegraphics[width=0.5\linewidth]{dfs_traversals.png}
\end{figure}

Though recursive is almost always far easier, the three forms of traversal can also be implemented iteratively with a stack. The different forms of iterative traversal with a stack-based implementation (in order of intuitiveness):

\begin{itemize}
    \item Preorder: at each current node, do the operation, push the right child onto the stack, set the left child as the current node, and repeat. If the current node is null, pop the current node from the stack. Repeat until the stack is empty.
    \item Inorder: at each current node, push that current node to the stack, set the current node to the left child, and repeat. If the current node is null, pop the current node from the stack, do the operation, and set the right node as current node. Repeat until the stack is empty.
    \item Postorder: at each current node, push its right child on the stack (mark as unvisited), then its left child (mark as unvisited), then itself (mark as visited), then set its left child, then pop the current node from the stack and repeat. In order to avoid pushing a right child twice, we keep track of a separate stack of booleans called visisted with the same structure and ordering as the node stack. If we pop a node from the stack that is visited, we only do the operation and don't push children, because those have been processed already.
\end{itemize}

A $O(n \log n)$ sorting algorithm for arrays is to first build a BST and then depth-first adding the values to the array like above.

\subsection{Backtracking}

\emph{Backtracking algorithms} (or rather, "backtracking metaheuristics") find solutions to some computational problems, notably constraint satisfaction problems, by incrementally building candidates to the solutions, and abandoning a candidate ("backtracking") as soon as they determine that the candidate cannot possibly be (part of) a valid solution. Backtracking can be applied only for problems which can have "partial candidate solutions" and an efficient test of whether it is invalid. If that's the case, backtracking is often much faster than brute-force enumeration of all complete candidates. Conceptually, the partial candidates can be represented as nodes of a search tree. Each partial candidate is the parent of the candidates that differ from it by a single extension step; the leaves of the tree are the partial candidates that cannot be extended any further. A backtracking algorithm traverses this search tree recursively, from the root down, in depth-first order.

\subsection{Sliding window algorithms}

Sliding window algorithms use two pointers demarcating a contiguous interval (of fixed or variable size) and slide those across an array. Two pointers algorithms use two pointers and generally only focus on the two elements pointed at. Sliding window algorithms are a subset of two pointers algorithms, though the window in between matters.

\subsection{Greedy algorithms}

A \emph{greedy algorithm} makes the locally optimal ("greedy") choice at each step, reducing each given problem into one smaller problem, i.e. it never reconsiders its choices. This is the main difference from dynamic programming, which is exhaustive and is guaranteed to find the solution. While greedy is optimal for some problems (e.g. classroom scheduling problem), for many problems, a greedy strategy does not produce an optimal solution but can heuristically yield a locally optimal solution faster (e.g. set covering problem). Greedy algorithms are a subset of \emph{approximation algorithms}.

\subsection{Decision problems}

An \emph{optimization problem} ask the minimum or maximum of something. A \emph{decision problem} has a yes-or-no answer. An optimization problem ("What is the shortest path?") can be repurposed as a decision problem ("Is there a path of length 5?"). All problems discussed in this subsection are decision problems. The class of (decision) problems of which the solution can be found in polynomial time (and, naturally, verified in polynomial time) is called P. For some problems, there is no known way to find an answer in polynomial time, but of which answers can be verified in polynomial time; these are in class NP (standing for "nondeterministic polynomial time"). P is a subset of NP. The \emph{P vs. NP problem} is a famous, unsolved computer science problem that asks whether every problem that is quick to verify is also quick to solve, i.e., whether P = NP.

A \emph{reduction} happens when you reformulate a problem you can't solve into one you do know how to solve. Problem A is reducible to problem B, if an algorithm for solving problem B efficiently could also be used as a subroutine to solve problem A efficiently. If so, solving A cannot be more complex than solving B. A problem is NP-hard if any problem in NP can be reduced, in polynomial time (because else the reduction may be the bottleneck), to that problem. As a consequence, a polynomial time solution for any one NP-hard problem gives us
a polynomial time solution for every problem in NP. A problem is NP-complete if it is both NP and NP-hard.

\subsubsection{Satisfiability problem}

The \emph{(boolean) satisfiability problem}, often abbreviated as SAT or B-SAT, is an NP-complete problem that asks whether there exists an assignment of a given propositional logic formula's boolean variables (i.e. consistently replacing them with \mintinline{java}{true} or \mintinline{java}{false}) that makes the formula evaluate to \mintinline{java}{true}. There's no efficient algorithm; you have to fill the entire truth table (in which each row is an assignment) in $O(2^n)$ time.

SAT is the first problem that was proven to be NP-complete (the Cook–Levin theorem), meaning that all problems in the complexity class NP are at most as difficult to solve as SAT.

\subsubsection{Traveling salesman problem}

The \emph{traveling salesman problem}, often abbreviated as TSP, is an NP-hard problem.

\subsubsection{Set cover problem}

\textsc{\textbf{Question}} Given a set of elements $\{1, 2, ..., n\}$ (the \emph{universe}), and a collection $S$ of $m$ subsets of universe whose union equals the universe, identify a smallest sub-collection of $S$ whose union equals the universe.

\textsc{\textbf{Answer}} The the set cover problem is an NP-hard problem. A greedy algorithm for polynomial time approximation chooses, at each step, the set that contains the largest number of uncovered elements

\subsection{Bit manipulation}

Different languages represent negative (32-bit) integers differently. There are three common ways to represent negative integers:

\begin{itemize}
    \item \textit{Sign-magnitude}; the \textit{most significant bit} (the first/left bit) is the \textit{sign bit}; if it's equal to \mintinline{java}{1}, the number is negative, if it's equal to \mintinline{java}{0}, the number is positive. The remaining bit represent the magnitude (or absolute value) of the number. It can only represent integers in the range $-(2^{n-1}-1)$ to $2^{n-1}-1$.
    \item \textit{Ones' complement arithmetic}; negative numbers are represented by the inverse (i.e. by flipping all bits) of the binary representations of their corresponding positive numbers. It can only represent integers in the range $-(2^{n-1}-1)$ to $2^{n-1}-1$.
    \item \textit{Two's complement arithmetic}; also uses a sign bit, but flipping the last bit to \mintinline{java}{1} doesn't simply negate the old value. While nonnegative numbers are simply given their unsigned binary representation, negative numbers are represented by taking the \textit{ones' complement} (by flipping all bits, i.e. NOT, or \mintinline{java}{~x}) of their (absolute) magnitude (i.e. their positive counterpart, in which the sign bit is \mintinline{java}{0}) and adding one, e.g. $6$ is \mintinline{java}{0110}, $-6$ is \mintinline{java}{1010}. The other way around, you can calculate the negative decimal from its binary representation (in which the sign bit is \mintinline{java}{1}) by calculating the (positive) decimal without the sign bit, and then subtracting the value of the sign bit (e.g. $2^31$ for a 32-bit integer). Two's complement decreases the complexity of bit manipulations and can express $-2^{n-1}$ to $2^{n-1}-1$ (one more number than the previous representations).
\end{itemize}

Java and Python use two's complement for negative integers. However, in Python, integers are objects that don’t use a fixed number of bit to store integers, but a variable number. Because they're objects, they also have some additional storage overhead. The \mintinline{python}{getsizeof(int)} in the \mintinline{python}{sys} module returns the number of bytes.

\section{Problems}

This sections summarizes algorithmic problems and their solutions. Sometimes, multiple solutions are possible, in which case they're ordered from good (enough) to better. For dynamic programming solutions, we generally only treat the bottom-up solution (unless the top-down solution is markedly different), but a top-down solution is always possible.

\subsection{Knapsack}

\textsc{\textbf{Question}} Given a set of items, each with a weight and a value, determine a subset of items that maximizes the total value while satisfying the weight constraint.

\textsc{\textbf{Answer}} Evaluating all subsets is $O(2^n)$. The greedy strategy of picking the most valuable thing at every step is suboptimal. 2D dynamic programming gives an optimal solution by solving the problem for smaller knapsacks and subsets of items. You row-wise fill a table of subproblems with as columns the max weights from $1$ up until the actual max weight\footnote{These can likely be integers, but if you have decimal weights, you need to have a finer column granularity, i.e. a column step size that accounts for this.}, and as rows subsets of items, with each row adding another item (the order doesn't matter, but it needs to be the same order as the order of the items in the weight vector). Every cell contains the solution considering the column's max weight and the row's subset of items, and is calculated as follows:

\begin{equation*}
    M_{i,j} = \max(M_{i-1,j}, \mathbf{v}_i + M_{i,j-\mathbf{w}_i})
\end{equation*}

subject to $j \geq \mathbf{w}_i$, where $M_{i,j}$ is the cell at row $i$ and column $j$ in matrix $M$, and $\mathbf{w}_i$ and $\mathbf{v}_i$ are the weight and value of the $i$\textsuperscript{th} item. In other words, we either the take the previous max value (i.e. without the item under consideration), or we take the value of the item plus the max value of the remaining space (as long as that's at least $0$).

\subsection{Linked List Cycle Detection}

\textsc{\textbf{Question}} Determine whether there is a cycle in a linked lists in $O(n)$ time and $O(1)$ space.

\textsc{\textbf{Answer}} A two pointers algorithm. Each iteration, the fast pointer moves by two and the slow moves by one; when they meet, there's a cycle. This is known as Floyd's fast and slow pointers algorithm or Floyd's tortoise and hare algorithm.\footnote{The algorithm is named after Robert W. Floyd, who was credited with its invention by Donald Knuth. However, the algorithm does not appear in Floyd's published work, and this may be a misattribution.}

\subsection{\href{https://leetcode.com/problems/contains-duplicate}{Contains Duplicate}}

\textsc{\textbf{Question}} Given an integer array, determine if any value appears at least twice in the array or if every element is distinct.

\textsc{\textbf{Answer}} Loop once through the array, adding each element to a hash set and checking whether it's already in there. $O(n)$ time, $O(n)$ space.

\subsection{\href{https://leetcode.com/problems/contains-duplicate-iii}{Contains Duplicate III}}

\textsc{\textbf{Question}} Same as Contains Duplicate II, but instead of equal values, we're looking for two numbers that differ at most by $t$, i.e. $||\mathbf{x}_i - \mathbf{x}_j|| \leq t$.

\textsc{\textbf{Answer}} Same as the answer to Contains Duplicate, but instead of a hash set, we use buckets with width $w = t + 1$. The bucket of any value $v$ can be determined with $\lfloor \frac{v}{w} \rfloor$. We return \mintinline{java}{true} if, for any value, there already exists a value in the same bucket, or whether there exists a value in the two adjacent buckets with values close enough together. Because there can be at most one value per bucket (or else we can return \mintinline{java}{true}), we can use a hash map for the buckets that maps bucket ID to its singular value. Of course, we remove values outside the window. $O(n)$ time, $O(\min(n,k))$ space.

\begin{minted}{python}
def containsNearbyAlmostDuplicate(nums, k, t):
    if t < 0:
        return False
    buckets = {}
    w = t + 1  # Increment by 1 to handle the range correctly
    for i in range(len(nums)):
        bucketIndex = nums[i] // w # Getting the bucket ID
        if bucketIndex in buckets:
            return True # There's already a value in the same bucket
        if bucketIndex - 1 in buckets and abs(nums[i] - buckets[bucketIndex - 1]) < w:
            return True # There's a value in a neigbouring bucket that's close enough
        if bucketIndex + 1 in buckets and abs(nums[i] - buckets[bucketIndex + 1]) < w:
            return True
        buckets[bucketIndex] = nums[i]
        if i >= k:
            del buckets[self.getID(nums[i - k], w)]
    return False
\end{minted}

There's also a $O(n \log n)$ answer that uses a BST as a data structure.

\subsection{\href{https://leetcode.com/problems/product-of-array-except-self}{Product of Array Except Self}}

\textsc{\textbf{Question}} Given an integer array \mintinline{java}{nums}, return an array with the value at each index \mintinline{java}{i} equal to the product of all the elements of \mintinline{java}{nums} except \mintinline{java}{nums[i]}, without using the division operation.

\textsc{\textbf{Answer}} Create an array \mintinline{java}{left} and make a pass over the input array from left to right, cumulatively multiplying the input array (so \mintinline{java}{left[i] = nums[i] * left[i-1]}), and do the same from right to left in array \mintinline{java}{right}. Then, each element in the output array is \mintinline{java}{left[i-1] * right[i+1]}. $O(n)$ time, $O(n)$ space.

A $O(1)$ space complexity optimization is to use the output array as the \mintinline{java}{left} array, and to only maintain an integer product for the right part:

\begin{minted}{python}
def productExceptSelf(nums):
    result = [0] * len(nums)
    result[0] = nums[0]
    for i in range(1, len(nums)):
        result[i] = nums[i] * result[i-1]
    prod = 1
    for i in reversed(range(1, len(nums))):
        result[i] = result[i-1] * prod;
        prod *= nums[i];
    result[0] = prod;
    return result
\end{minted}

\subsection{\href{https://leetcode.com/problems/longest-consecutive-sequence}{Longest Consecutive Sequence}}

\textsc{\textbf{Question}} Given an unsorted array of integers, return the length of the longest consecutive elements sequence in $O(n)$ time. A consecutive sequence is a sequence where each subsequent element is one larger than the previous element.

\textsc{\textbf{Answer}} Put all numbers in a hash set (for $O(1)$ lookup), loop through all the numbers (either through the array or the set, the order doesn't matter), and for each number \mintinline{java}{num}, check if \mintinline{java}{num-1} is in the set. If it is, this number is not the start of a consecutive sequence and we skip. If it isn't, count the length of the consecutive sequence that starts with this number by iteratively checking whether the increment is in the set. Return the maximum of such a length.

\begin{minted}{python}
def longestConsecutive(nums):
    longest_streak = 0
    num_set = set(nums)

    for num in num_set:
        if num - 1 not in num_set:
            current_num = num
            current_streak = 1

            while current_num + 1 in num_set:
                current_num += 1
                current_streak += 1

            longest_streak = max(longest_streak, current_streak)

    return longest_streak
\end{minted}

This is $O(n)$ time and space, because each element is processed a constant number of times; the inner while loop only runs for each consecutive sequence.

\subsection{\href{https://leetcode.com/problems/3sum}{3Sum}}

\textsc{\textbf{Question}} Given an integer array, return all the distinct triplets (\mintinline{java}{[nums[i], nums[j], nums[k]]}, at distinct indices) that sum to zero.

\textsc{\textbf{Answer}} This is the solution to Two Sum repeated for every number (though you only have to check to the right of every number as an optimization).

\begin{minted}{python}
public List<List<Integer>> threeSum(int[] nums) {
    Set<List<Integer>> triplets = new HashSet<>();

    for(int i = 0; i < nums.length; i++) {
        Set<Integer> numset = new HashSet<>();
        for(int j = i+1; j < nums.length; j++) {
            if(numset.contains(-(nums[i] + nums[j]))) {
                List<Integer> list = Arrays.asList(nums[i], nums[j], -(nums[i] + nums[j]));
                Collections.sort(list);
                triplets.add(list);
            }
            numset.add(nums[j]);
        }
    }

    return new ArrayList(triplets);
}
\end{minted}

\subsection{\href{https://leetcode.com/problems/trapping-rain-water}{Trapping Rain Water}}

\textsc{\textbf{Question}} Given an array of non-negative integers representing an elevation map where the width of each bar is 1, compute how much water it can trap after raining.

\textsc{\textbf{Answer}}

\begin{itemize}
    \item Create two arrays, one called \mintinline{java}{leftMax} for storing the max height from the beginning up to \mintinline{java}{i}, and one called \mintinline{java}{rightMax} for storing the max height from the end up to \mintinline{java}{i}, and initialize them with two loops (front-to-back for \mintinline{java}{leftMax} and back-to-front for \mintinline{java}{rightMax}). In a final loop through the array, you calculate the volume of water for each individual bar using $\min(\mathtt{leftMax[i]}, \mathtt{rightMax[i]}) - \mathtt{heights[i]}$ and add it to a sum.
    \item Create a stack for storing the indices of increasingly longer bars (from the top to the bottom) seen so far. Loop over the array; for each current element, while the current bar is longer than that at the top of the stack, we are sure that the bar at the top of the stack is bounded by the current bar and a previous bar in the stack; hence, pop it and add the resulting trapped water, calculated as the distance (the current index minus the new top of the stack minus 1) multiplied by the bounded height ($\min(\mathtt{heights[i]}, \mathtt{heights[stack.peek()]}) - \mathtt{heights[popped]}$). After all bars smaller than the current one have been popped, push the current index to top of the stack and continue.
    \item \textit{Two pointers}. Initialize \mintinline{java}{left} and \mintinline{java}{right} pointers at both ends of the array, and two integers \mintinline{java}{leftMax} and \mintinline{java}{rightMax}. Move them towards each other; in each iteration, process and move the pointer pointing to the smallest bar: if that bar is larger than (or equal to) the corresponding max, update the max, and if it is smaller, add the corresponding max minus the current height to the sum.
\end{itemize}

\subsection{\href{https://leetcode.com/problems/longest-repeating-character-replacement}{Longest Repeating Character Replacement}}

\textsc{\textbf{Question}} Find the longest \emph{substring} (i.e. contiguous characters) of the same uppercase characters in a string of uppercase characters, in which you can convert at most $k$ characters to another character (i.e. you have $k$ wildcards).

\textsc{\textbf{Answer}}

\begin{itemize}
    \item \textit{Two-pointers sliding window}. For each possible character, you slide through the array with the right pointer, only advancing the left pointer when the number of "non-loop characters" becomes larger than $k$. Maintain a max of the window length and return it. $O(nm)$ time, $O(m)$ space, where $m$ is the number of distinct possible character values.
    \item Actually hard. \textit{Two-pointers sliding window}. The right pointer keeps moving, and max'ing the length of the current window. At every iteration, a hashmap of character frequencies currently in the window\footnote{In Java, the map can be a simple \mintinline{java}{int[26]} indexed with \mintinline{java}{s.charAt(q) - 'A'}} is updated by incrementing the character at the right pointer. Also, \mintinline{java}{maxFrequency} keeps track of the maximum of any such frequency of any character in any window seen so far, and we update that as well. If \mintinline{java}{right - left + 1 - maxFrequency <= k} the window is an invalid substring, in which case the left pointer is incremented (and the corresponding frequency decremented); otherwise, the loop continues. Note that both the window size (\mintinline{java}{right - left + 1}) as well as \mintinline{java}{maxFrequency} never decreases, which doesn't matter, because we're only interested in larger windows. $O(n)$ time, $O(m)$ space.
\end{itemize}

\subsection{\href{https://leetcode.com/problems/find-minimum-in-rotated-sorted-array}{Find Minimum in Rotated Sorted Array}}

\textsc{\textbf{Question}} Given a sorted but rotated array (e.g., \mintinline{java}{[4,5,6,7,0,1,2]} is rotated 4 times), return the minimum element of this array.

\textsc{\textbf{Answer}} \textit{Adapted binary search}. While \mintinline{java}{nums[left] > nums[right]}, we continue searching in the half where the "breaking point" is (where the original last/max elements is adjacent to the original first/min element), after which we return the element at the left pointer. $O(\log n)$ time, $O(1)$ space. Code:

\begin{minted}{python}
def find_min(nums):
    p, q = 0, len(nums) - 1
    while nums[p] > nums[q]:
        h = (p + q) // 2
        if nums[h] >= nums[q]:
            p = h + 1
        else:
            q = h
    return nums[p]
\end{minted}

\textsc{\textbf{Question}} Given the root of a binary tree, return the maximum path sum of any non-empty path. The path can be any connected sequence of nodes. Node values can be negative.

\textsc{\textbf{Answer}} Actually medium. Recurse from the root. You return a slightly different value, namely the maximum path sum if the node is the end/beginning of a path. $O(n)$ time, $O(n)$ (because of the call stack).

\begin{minted}{python}
maxPath = -float("inf")
gainFromSubtree(root)
return maxPath

def gainFromSubtree(node): # Post order traversal. Return the max sum for a path starting at the root of subtree
    if not node:
        return 0

    gainFromLeft = max(gainFromSubtree(node.left), 0) # Use max, because if gain is negative we can ignore it
    gainFromRight = max(gainFromSubtree(node.right), 0) # Idem
    maxPath = max(maxPath, gainFromLeft + gainFromRight + node.val)
    return max(gainFromLeft + node.val, gainFromRight + node.val)
\end{minted}

\subsection{\href{https://leetcode.com/problems/house-robber}{House Robber}}

\textsc{\textbf{Question}} Given an integer array representing amounts of money stashed in houses along a street, return the maximum amount of money you can rob without robbing two adjacent houses. The values are nonnegative.

\textsc{\textbf{Answer}} Bottom-up 1D dynamic programming. We need to only memorize the last two answers as mentioned in Section \ref{dynamic_programming}, so we don't need a full array. Loop front-to-back through the array, at every position updating \mintinline{java}{maxLastIncluded = nums[i] + maxLastExcluded} and \mintinline{java}{maxLastExcluded = Math.max(maxLastIncluded, maxLastExcluded)} (and using a temporary variable to store the first variable you overwrite, because you're sort of swapping numbers; this code is just to show the intuition). $O(n)$ time, $O(1)$ space. A very concise Python implementation:

\begin{minted}{python}
def rob(nums):
    prev, curr = 0, 0
    for n in nums:
        prev, curr = curr, max(curr, prev + n)
    return curr
\end{minted}

\subsection{\href{https://leetcode.com/problems/house-robber-iii}{House Robber III}}

\textsc{\textbf{Question}} Same as House Robber, but the street is a binary tree, and you're given the \mintinline{java}{root}.

\textsc{\textbf{Answer}} A pretty straightforward adaptation of the answer to House Robber:

\begin{minted}{python}
def rob(root):
    def returnTupleWithMaxLootThatIncludesAndExcludesThis(node):
        if not node:
            return (0, 0)
        left = returnTupleWithMaxLootThatIncludesAndExcludesThis(node.left)
        right = returnTupleWithMaxLootThatIncludesAndExcludesThis(node.right)
        robThis = node.val + left[1] + right[1] # If we rob this node, we cannot rob its children
        dontRobThis = max(left) + max(right) # Else we can rob either of its children
        return [robThis, dontRobThis]

    return max(returnTupleWithMaxLootThatIncludesAndExcludesThis(root))
\end{minted}

\subsection{\href{https://leetcode.com/problems/house-robber-iv}{House Robber IV}}

\textsc{\textbf{Question}} Same as House Robber, but instead of maximizing the total reward, minimize the maximum amount stolen from any single house while ensuring that at least $k$ houses are robbed.

\textsc{\textbf{Answer}} In Section \ref{binary_search}, we said that \textit{problems that require minimizing the maximum or maximizing the minimum often suggest a binary search approach on the range of possible values for this maximum or minimum, respectively}. Demarcate the search space of the possible values for the maximum loot (from any single house) by taking the minimum and maximum of all element values in the array. Then, use binary search to determine the minimum possible maximum loot of any burglary spree involving at least $k$ houses. At every iteration, you take the middle of the current search space and check whether that value can be the maximum of $k$ non-contiguous houses. This check is done by iterating through the list of house values and greedily selecting houses that have at most the value to be checked (of course skipping the next house each time we choose one). Once we rob $k$ houses, it means the current middle value is a valid maximum, in which case we move the binary search range to the left; else we move it to the right. $O(n \log m)$ time (where $m$ is the size of the search space), $O(1)$ space.

Also do this for the palindromes of even length. Returns the largest palindrome found. $O(n)$ time, $O(1)$, so it's space complexity is superior than the DP solution, but in practice it's also faster, because the DP solutions always checks $n^2$ pairs, while this solutions abandons early.

\subsection{\href{https://leetcode.com/problems/maximum-subarray}{Maximum Subarray}}

\textsc{\textbf{Question}} Given an integer array nums, find the subarray (i.e. contiguous) with the largest sum, and return that sum.

\textsc{\textbf{Answer}} \textit{Kadane's algorithm}. Loop through the array while maintaining a current sum and a maximum sum, at each element you first take the max of the current sum and $0$ (because as soon as the sum is negative you can throw it away and start over), then add the current element to the sum and take the maximum of that and the maximum found so far. Return the maximum sum. $O(n)$ time, $O(1)$ space.

\begin{minted}{python}
max = -infinity # Set to 0 if empty subarrays are allowed
sum = 0
for x in numbers:
	sum = max(x, sum + x) # Set to max(0, sum + x) if empty subarrays are allowed
	max = max(max, sum)
return max
\end{minted}

Q follow-up: Use a divide-and-conquer algorithm to solve the problem (even though it has a worse time and space complexity).

A follow-up: Write a recursive function that takes the left and right inclusive bounds. Recurse in the left and right halves and get the sums of optimal subarrays contained in both halves. Then, loop from the middle to the start, keep summing and maxing the sum with a maximum. Do the same from the middle to the end of the array, and sum the two. Return the max of the left half, the right half, and the max containing the middle.

\subsection{\href{https://leetcode.com/problems/maximum-product-subarray}{Maximum Product Subarray}}

\textsc{\textbf{Question}} Given an integer array nums, find a subarray (i.e. contiguous) with the largest product, and return that product.

\textsc{\textbf{Answer}}

\begin{itemize}
    \item \textit{Dynamic programming}. Loop through the array while maintaining a global maximum, a current maximum, and a current minimum value. At each element, the current maximum is updated with the max of: the current element, the current maximum times the current element, or the current minimum times the current element. The current minimum is updated with the minimum of these same three values. At the end of each loop, update the global maximum by maxing it with the current maximum, and return the global maximum after the loop. In this way, 0s are handled correctly because they "reset" the current maximum and minimum, every pair of negative numbers is handled correctly because the current minimum retains the product of the elements between them, and every odd number of negatives is handled correctly because the current maximum is reset to the element after it. $O(n)$ time, $O(1)$ space.
    \item \textit{Emiel's algorithm}. Genius theoretical computer scientist Emiel Bos has the following interpretation of the problem. You keep a global maximum and a current product and loop through the array, maxing the current product with the global maximum. 0s break up the array as if the subsequent element were the start of the array; just reset the current product and continue. Apart from 0s, there are negative numbers. Zero or an even number of negative numbers means the maximum product subarray is the whole array; the current product will capture it's max. If there is an uneven number of negatives, then the maximum product subarray either starts after the first such negative (and ends at the array end), or ends before the last such negative (and starts at the array start). If the subarray is the latter, the current maximum product will capture it. For capturing the former, you will need to maintain a separate product that starts multiplying after the first negative number (also the first after a 0), and keep maxing that with the global maximum. $O(n)$ time, $O(1)$ space.
\end{itemize}

\subsection{\href{https://leetcode.com/problems/longest-increasing-subsequence}{Longest Increasing Subsequence}}

\textsc{\textbf{Question}} Given an integer array, return the length of the longest strictly increasing subsequence (i.e. not contiguous).

\textsc{\textbf{Answer}}

\begin{itemize}
    \item \textit{Bottom-up 1D dynamic programming.} Initialize an array with length equal to the input array and all elements equal to 1. \mintinline{java}{dp[i]} should represent the length of the longest increasing subsequence that ends with the \mintinline{java}{i}th number. Loop through the array, at every position you do a nested loop up until that position to find the element with the largest subsequence of which the corresponding number is smaller than the current one. $O(n^2)$ time, $O(n)$ space.
    \item \textit{Building a subsequence.} Actually hard. Loop through the input array while building up a strictly increasing subsequence. At each number, if it is greater than the largest number in the subsequence, add it. Otherwise, perform a linear smallest-to-largest scan through the subsequence and replace the first element that is greater than the number. This opens the door for elements that are greater than the inserted number but less than the replaced number. This algorithm does not always generate a valid subsequence of the input, but the length of the subsequence will be correct, because the length only changes when a new element is larger than the largest element in the subsequence, and this largest value will only drop when you encounter a number that is smaller only to this last number in the subsequence. Replacing earlier elements in the subsequence will increase the chance of that happening. $O(n^2)$ time, $O(n)$ space.
    \item \textit{Building a subsequence with binary search.} Instead of a linear scan, use binary search to find the element-to-replace in the above solution. $O(n \log n)$ time, $O(n)$ space.
\end{itemize}

\subsection{\href{https://leetcode.com/problems/unique-paths}{Unique Paths}}

\textsc{\textbf{Question}} Given two integers $m$ and $n$, return the number of possible unique paths from the top-left corner of a $m \times n$ grid to the bottom-right corner when you can only move right and down.

\textsc{\textbf{Answer}}

\begin{itemize}
    \item \textit{Bottom-up 2D dynamic programming.} Actually easy. Initialize a two-dimensional $m \times n$ integer array with \mintinline{java}{1}s in the first row and the first column, where \mintinline{java}{dp[i][j]} indicates the number of unique paths to those coordinates, loop through all the other cells and set \mintinline{java}{d[i][j] = d[i-1][j] + d[i][j-1]}. Return \mintinline{java}{dp[m-1][n-1]}. $O(nm)$ time, $O(nm)$ space.
    \item \textit{Math}. Actually really hard. There are $h+v$ moves to do from start to finish, $h=m-1$ horizontal moves and $v=n-1$ vertical ones. One could choose when to move to the right (to define $h$) and that will fix vertical ones, or one could choose when to move down (to define $v$) and that will fix horizontal ones.
    \begin{equation}
        \binom{h}{h+v} = \binom{v}{h+v} = \frac{(h+v)!}{h!v!} = \frac{(m+n-2)!}{(m-1)!(n-1)!}
    \end{equation}
    The best known algorithm to compute factorial function is by Peter Borwein. The idea is to express the factorial as a product of prime powers. $O((m+n)(\log(m+n)\log\log(m+n))^2)$ time, $O(1)$ space.
\end{itemize}

\subsection{Longest Common Substring}

\textsc{\textbf{Question}} Determine the longest common \emph{substring} (i.e. contiguous characters) between two given strings.

\end{minted}

In the second loop, set an element to \mintinline{java}{0} if either its row or column has a \mintinline{java}{0} as first element. $O(n)$ time, $O(1)$ space.

\subsection{\href{https://leetcode.com/problems/number-of-1-bits}{Number of 1 Bits}}

\textsc{\textbf{Question}} Given a positive integer, return the number of \mintinline{java}{1} bits in its binary representation (also known as the \textit{Hamming weight}).

\textsc{\textbf{Answer}}

\begin{itemize}
    \item \textit{Loop a mask}. Initialize a mask to the value $1$ (which is only one \mintinline{java}{1} bit in the rightmost position), and repeat 32 times: check whether its AND with the integer is not zero, and shift once to the left.
    \item Slightly simpler and faster. For any number $n$, doing a bit-wise AND of $n$ and $n-1$ flips the least-significant \mintinline{java}{1}-bit to \mintinline{java}{0}. Simply count the number of times you can do that without the integer becoming $0$.
\end{itemize}

\subsection{\href{https://leetcode.com/problems/counting-bits}{Counting Bits}}

\textsc{\textbf{Question}} Given an integer $n$, return an array of length $n + 1$ such that the value at each index is the number of \mintinline{java}{1}'s in the binary representation of the index.

\textsc{\textbf{Answer}}

\begin{itemize}
    \item \textit{Bottom-up 1D dynamic programming}. Create the array and loop: at each index, you use the bit-wise AND of $n$ and $n-1$ trick to flip the least-significant \mintinline{java}{1}-bit in the index number, and use the result to index into the array, increment the found value by 1, and store that.
    \item \textit{Bottom-up 1D dynamic programming}. There's another transition function we can use. We can also index into the array using half the current index; \mintinline{java}{x/2} is equal to \mintinline{java}{x >> 1}. This number will only differ by one from the current index if the current index is odd, so we need to check that with \mintinline{java}{x % 2}, which is equal to \mintinline{java}{x & 1}, so each index is filled with \mintinline{java}{dp[i] = dp[i >> 1] + (i & 1)}.
\end{itemize}

\subsection{\href{https://leetcode.com/problems/reverse-bits}{Reverse Bits}}

\textsc{\textbf{Question}} Given a 32-bit integer, reverse its bits.

\textsc{\textbf{Answer}}

\begin{itemize}
    \item \textit{Bit by bit}. Initialize a result integer with \mintinline{java}{0} and a \mintinline{java}{power} integer with \mintinline{java}{31}. In a \mintinline{java}{while(n != 0)} loop, take the rightmost bit using \mintinline{java}{while(n & 1)} (or \mintinline{java}{while(n % 2)}), shift this bit \mintinline{java}{power} bits to the left, add it to the result integer, shift the input integer 1 to the right and decrement power by 1. $O(1)$ time, $O(1)$ space.
    \item \textit{Divide and conquer}. Switch the two block of 16 bits, the four blocks of 8 bits, the eight block of 4 bits, the sixteen block of 2 bits, and every pair of bits. $O(1)$ time, $O(1)$ space.

    \begin{minted}{python}
def reverseBits(self, n: int) -> int:
    n = (n >> 16) | (n << 16)
    n = ((n & 0xFF00FF00) >> 8) | ((n & 0x00FF00FF) << 8)
    n = ((n & 0xF0F0F0F0) >> 4) | ((n & 0x0F0F0F0F) << 4)
    n = ((n & 0xCCCCCCCC) >> 2) | ((n & 0x33333333) << 2)
    n = ((n & 0xAAAAAAAA) >> 1) | ((n & 0x55555555) << 1)
    return n
    \end{minted}
\end{itemize}

\subsection{\href{https://leetcode.com/problems/sum-of-two-integers}{Sum of Two Integers}}

\textsc{\textbf{Question}} Given two integers \mintinline{java}{a} and \mintinline{java}{b}, return their sum without using the operators \mintinline{java}{+} and \mintinline{java}{-}.

\textsc{\textbf{Answer}} To reduce the number of cases to handle, first recursively call the same function with the operands flipped if \mintinline{java}{abs(a) < abs(b)}, so that we're sure the first operand has a bigger or equal magnitude than the second. In that case, the sign of \mintinline{java}{a} determines the sign of the result, so store that (as either \mintinline{java}{-1} or \mintinline{java}{1}). There are two cases remaining:

\begin{itemize}
    \item If \mintinline{java}{a * b >= 0}, we calculate the sum of their absolutes. XORing two integers results in their sum without taking carry into account. We can get the carry with \mintinline{java}{(abs(a) & abs(b)) << 1}. Now, we just need the sum of \mintinline{java}{abs(a) ^ abs(b)} and this carry, which is the same problem again (summing two integers), so we iteratively calculate the XOR of the previous XOR (the previous sum without carry) and the previous carry, until the new carry is zero, at which point we have the answer.
    \item If \mintinline{java}{a * b < 0}, we calculate the difference of their absolutes. XORing two integers results in their difference without taking borrow into account, so this is very similar to summing, but instead of carry we calculate borrow as \mintinline{java}{((~abs(a)) & abs(b)) << 1}.
\end{itemize}

Return the result multiplied by the sign.
